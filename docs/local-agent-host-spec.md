# Foreman Local Agent Host - Desktop CU Driver Channel

Build-ready spec from a design + red-team workflow (4 architectures -> 3 channel red-teamers -> synthesis; 29 attacks, all closed). The Local Agent Host lets a LOCAL AI agent pilot the desktop
THROUGH Foreman's audited/confined/panic-killable path, over a local-only channel (never the network; the MCP
server stays barred from desktop). The audit/confine/inject/panic spine is REUSED UNCHANGED.

## Summary

Chosen: an owner-only DUPLEX named pipe with a TWO-HOP topology - Foreman LAUNCHES a Foreman-signed Pilot shim (Foreman.CuPilot, HOP A, reusing DesktopCuController's launch-bound VerifyClientIdentity + PinBinaryAtRest + handshake VERBATIM), and the operator's arbitrary AI agent connects to that shim over a second owner-only pipe (HOP B). This grafts Design 3's spoof-resistant launch-rooted identity (the only model the red-team certifies safe against a same-user attacker 'for free') onto the dangerous broker-reaching hop, while still letting the operator run their own agent. It folds in Design 1's structural wire contract (a distinct DriverSubmit record that physically cannot carry hwnd/epoch/ByHarness) and Design 4's authorization hardening (per-action default-Held, '*' never authorizing Desktop). Design 2's loopback TCP listener is rejected (four HIGH/CRITICAL findings: racy fail-open peer attribution, bind-correctness-dependent non-reachability, DNS-rebind surface). All 26 red-team findings are closed: the two DPAPI-nonce HIGHs and the two enrolled-image-swap CRITICALs are eliminated by never using a readable nonce file and keeping launch-bound identity on HOP A; the Critical 'presence off by default' and HIGH 'one tap covers bind+enroll+resume' are closed by a force-presence + fresh-tap + (action,detail)-keyed cache; the forgeable-bind HIGH by a probe-token-minted CuWindowRef + bind-target gating; the connect-race HIGH by gating every connection and re-looping the single instance; the wildcard-to-desktop HIGH by excluding '*' from Desktop; the HUD findings by a per-input adversarial occlusion test on an App-defined turn clock; the canary-window Medium by per-round-trip DetectTamper. The entire audit/confine/inject/panic spine (CuBroker, the hardened sidecar injector, CuSharedPanicFlag, CuDesktopPanicFloor, PanicController, the to-be-built DesktopCuExecutor+CuExecutorPump) is reused UNCHANGED; the only new code is the two-pipe channel, the in-process submit pump, two WeakeningActions, the presence-hardening overload, the probe-token bind seam, and the per-input HUD ack. Build order: L0 core seams + structural contract, L1 presence hardening, L2 unforgeable+target-gated bind, L3 launch-bound shim (HOP A), L4 agent channel (HOP B), L5 in-process pump + default-Held + per-round-trip canary, L6 per-input adversarial HUD ack + panic wiring - each with a green xUnit or on-device gate, input never firing before the audit and confinement spine exist.

## Chosen architecture

PRIMARY TRANSPORT: an owner-only DUPLEX named pipe (no TCP socket), grafted from Design 3 (the "Pilot Sidecar") as the security spine, with Design 1's structural wire-contract discipline and Design 4's authorization hardening folded in. The chosen design is NOT any single candidate verbatim; it is Design 3's Foreman-LAUNCHES-the-driver model (the only one the red-team certifies as same-user-spoof-resistant "for free") + Design 1's distinct DriverSubmit record (so the wire contract is structurally incapable of carrying hwnd/epoch/ByHarness) + Design 4's "* never authorizes Desktop" + per-action default-Held posture.

WHY Design 3's launch model wins (decisive over 1/2/4): the operator-launches-the-host model in Designs 1/2/4 cannot reuse the sidecar's strongest gate (PID==launched, parent==Foreman, image==Foreman's signed exe) because the driver is an arbitrary third-party binary. Red-team findings prove every substitute they reach for is same-user-defeatable: DPAPI-CurrentUser nonce files are readable by any same-user process (two HIGH findings); enrolled-image-path Gate 2 falls to an at-rest binary swap on user-writable paths (two CRITICAL findings); a same-user process can BE the enrolled image by launching+injecting it. Design 3 sidesteps all of this: Foreman launches a Foreman-SIGNED broker-shim child (Foreman.CuPilot) it can PinBinaryAtRest + signer-verify + PID-pin, exactly the proven DesktopCuController spine. The operator's own arbitrary AI agent connects to THAT shim over a SECOND owner-only pipe; the launch-bound, spoof-resistant identity gates the dangerous broker-reaching hop, and the arbitrary agent never touches the broker channel directly (the red-team's own recommended graft for 1/2/4). So we keep "the operator runs their own agent" while keeping Design 3's launch-rooted identity on the dangerous edge.

WHY a pipe over a socket (rejects Design 2): Design 2's second loopback Kestrel listener drew FOUR HIGH/CRITICAL findings - it is a real network listener whose non-reachability rests on bind-correctness (a future ListenAnyIP/reverse-proxy regression re-exposes it) rather than an OS object boundary; its LoopbackPeer 5-tuple attribution is racy and FAILS OPEN on Unattributed (Critical); and it re-imports the DNS-rebind/forged-request surface the pipe designs delete by construction. A named pipe has no port, no bind address, no SMB-grantable ACE beyond the single interactive-user SID, and yields the kernel's live client PID via GetNamedPipeClientProcessId with no table race. Design 2 is rejected outright.

## Channel design (two-hop named pipe)

TWO-HOP PIPE TOPOLOGY (the graft):

HOP A - Foreman.App <-> Foreman.CuPilot (the trusted shim). Foreman LAUNCHES Foreman.CuPilot.exe (new project, mirrors Foreman.CuSidecar: WinExe, net10.0-windows, asInvoker, medium-IL, staged under AppContext.BaseDirectory\cu-pilot\) exactly as DesktopCuController launches the injector sidecar. PilotChannelController (new, App-side) clones DesktopCuController's launch+handshake+lifecycle VERBATIM: CreateOwnerOnlyDuplexPipe (single instance, owner-SID-only DACL), per-launch random name "foreman-pilot-<guid:N>", nonce via argv (HOP A only - argv-scrape is moot here because Gate 2's PID/parent/signed-image pin rejects any non-launched process; the red-team confirms Design 3's launched-PID pin holds even with a scraped nonce). It REUSES SidecarIntegrity.Verify, PinBinaryAtRest (the shim is Foreman's own signed binary - fully pinnable, unlike a third-party host), VerifyClientIdentity (PID==launched, parent==Foreman, QueryFullProcessImageNameW==pinned exe, re-hash at connect), CuHandshake challenge/response, CuJson.ResponseMac, connect+per-request deadlines, reap-on-any-exit, and adds KillPilotNow (mirrors KillSidecarNow) to the panic kill set. The shim's ONLY outbound effect is to relay a DriverSubmit to the App; it holds NO MMF handle and has no ExecuteAction verb.

HOP B - Foreman.CuPilot <-> the operator's arbitrary AI agent. The Pilot shim hosts a SECOND owner-only duplex pipe the operator's local agent connects to. This hop carries the LEAST trust: the agent is arbitrary, so the shim authenticates it only by owner-only DACL (same-user) + a per-session nonce the OPERATOR transfers to their agent at bind time (presence-gated, see driverAuthDesign) + challenge/response, and treats the agent purely as a proposer. Crucially, the shim does NOT forward the agent's identity claims; it stamps a fixed authenticated id "local-agent-host" on every DriverSubmit it relays to the App over HOP A. A compromised agent on HOP B can therefore only PROPOSE; it can never reach the broker except through the launch-bound, signed shim on HOP A.

WHO CAN CONNECT: HOP A - only the process Foreman spawned, kernel-PID-pinned + signed-image-verified (a same-user attacker is not the launched PID, not parented to Foreman, not the pinned signed image). HOP B - one same-user agent that holds the operator-transferred per-session nonce; on ANY failed handshake the shim DISCONNECTS and LOOPS WaitForConnectionAsync on the SAME server instance (closes red-team finding #1: never treat "connected on the single instance" as identity; a racing impostor is rejected and re-looped so the real agent still gets the slot; never abandon the launch). Connect on a fresh per-session name by a non-enrolled process raises HIGH (it knew the name => leak/race).

LIFECYCLE: off by default behind CuDriverHostEnabled (mirrors CuDesktopEnabled). Arming requires an enrolled presence credential (see invariants). Both pipes are single-instance; both re-create the server instance on any failed-handshake disconnect so a squat cannot consume the slot. Panic tears down both pipes AND KillPilotNow's the shim (which drops HOP B), so resume starts from a clean slate requiring a fresh operator launch+bind.

## Driver authentication (4 layers)

HOW A MALICIOUS SAME-USER PROCESS IS EXCLUDED - four layers, reusing Foreman's hardened patterns, fail-closed at every gate:

LAYER 1 (HOP A, the dangerous broker-reaching hop) - LAUNCH-BOUND IDENTITY, reused UNCHANGED from DesktopCuController.VerifyClientIdentity. The Pilot shim is Foreman's OWN signed binary, launched by Foreman. The connecting client must be GetNamedPipeClientProcessId == the PID Process.Start returned, NtQueryInformationProcess parent == Environment.ProcessId, QueryFullProcessImageNameW == the PinBinaryAtRest-locked signed exe, re-hashed at connect (TOCTOU-closed). A same-user attacker CANNOT be this process: not the launched PID, not parented to Foreman, not the pinned signed image. This is the gate Designs 1/2/4 had to throw away and could not replace; HOP A keeps it. The argv nonce is unscrapable-in-effect here because Layer 1 pins the PID regardless (red-team finding #6 confirms the launched-PID pin is the sole-but-sufficient defense even with a scraped nonce; we additionally close finding #1's race with the disconnect-and-reloop).

LAYER 2 (HOP B, the arbitrary-agent hop) - OPERATOR-TRANSFERRED SESSION SECRET, not a readable file. The shim mints a per-session nonce and hands it to the operator's agent ONLY via a channel the operator chooses at bind time (an inherited handle if the operator lets Foreman launch the agent, OR a one-time secret the operator pastes into their agent's config under the presence-gated bind transaction). It is NEVER written to a DPAPI-CurrentUser file (closes both DPAPI HIGH findings: DPAPI-CurrentUser gives ZERO same-user separation; any sibling can Unprotect it). Challenge/response over CuHandshake proves knowledge without the secret crossing the wire; every HOP B frame is CuJson.ResponseMac-authenticated. A same-user attacker without the operator-transferred secret cannot answer the challenge.

LAYER 3 - OPERATOR AUTHORIZATION as the root, hardened per Design 4 + red-team. The shim's relayed DriverSubmit lands in CuBroker.SubmitAsync with ByHarness = "local-agent-host" (App-set, never agent-supplied). CanDrive must return true for it - AND the "*" wildcard is treated as NOT authorizing Desktop modality (closes the HIGH wildcard finding present in Designs 1/2/3): a desktop driver must be an explicitly enumerated, operator-enrolled id, never "any". Enrolling that id is a presence-gated WeakeningAction (see invariants), bound to the presence credential so editing sealed settings alone cannot self-enroll (closes the seal-bypass Medium finding).

LAYER 4 - DACL + single-instance + sticky session. Owner-only pipe DACL excludes every other USER (kernel-checked, no remote SMB principal). maxNumberOfServerInstances:1 + disconnect-and-reloop on failed handshake (finding #1). The authenticated connection IS the session; no stateless per-request re-auth a spoofer could race.

NET: a medium-IL same-user attacker must simultaneously BE Foreman's launched signed shim (Layer 1, impossible) OR hold the operator-transferred session secret (Layer 2) AND be the operator-enrolled, presence-bound driver id (Layer 3). The only residual is the operator deliberately binding a malicious agent - bounded by per-action approval, one-window confinement, HUD, and panic (documented, not closed by auth - same class as choosing which harness to allow-list over MCP).

## Invariants (INV-11 .. INV-20)

- INV-11 (desktop driver is a launch-rooted named pipe, never a socket, never the network): the Local Agent Host reaches the broker ONLY through Foreman's own LAUNCHED, signed, PID-pinned Pilot shim over an owner-only duplex named pipe. There is no TCP listener, no port, no bind address, no SMB-grantable ACE beyond the single interactive-user SID; the network MCP server continues to hard-reject modality=desktop (ForemanMcpTools.cs:1337). INV-7 becomes an OS object-model property, not a bind-correctness invariant a future commit can break.
- INV-12 (the agent PROPOSES, never acts, and its wire contract is structurally incapable of acting): the only effect the Local Agent Host can have is an in-process CuBroker.SubmitAsync relayed by the shim with App-set ByHarness='local-agent-host', App-set Modality=Desktop, and broker-stamped hwnd/epoch. The wire frame is a distinct DriverSubmit{actionId,verb,args,justification} record with NO hwnd/epoch/modality/ByHarness fields (Design 1's structural exclusion), so even a buggy forwarder cannot carry them; the App builds the CuAction from scratch. The agent has no bind, resume, enroll, claim, execute, or injector-handle path.
- INV-13 (driver identity is launch-bound on the dangerous hop, operator-transferred-secret on the agent hop; never a same-user-readable file): HOP A (shim<->App) reuses VerifyClientIdentity (PID==launched, parent==Foreman, signed pinned image, re-hash at connect) UNCHANGED. HOP B (agent<->shim) authenticates by owner-only DACL + a per-session secret the OPERATOR transfers at bind time (inherited handle or presence-gated one-time transfer), never a DPAPI-CurrentUser file. On ANY failed handshake the server DISCONNECTS and re-loops WaitForConnectionAsync on the same single instance - a connect race cannot consume the slot or be mistaken for identity.
- INV-14 ('*' never authorizes Desktop; desktop authorization is an explicitly enumerated, presence-bound, operator-enrolled id): CanDrive (or a desktop pre-submit check) treats the ['*'] wildcard driver set as NOT authorizing Desktop modality. A desktop driver must be an enumerated id enrolled under a live presence tap, and the enrollment record is bound to the presence credential so editing sealed settings.json alone cannot grant it. A broker invariant test asserts ['*'] + Desktop submit => Held/Blocked, never Approved.
- INV-15 (propose-not-act default; auto-grant is bounded, presence-armed, and idle-revoking): desktop actions from the Local Agent Host land HELD for operator approve/deny even on auditor Allow (CuDesktopAutoGrant default OFF), so a compromised channel cannot act without a human in the loop. If auto-grant is opted in, its TTL is capped FAR below the 300s presence cap (30-60s) plus a hard per-grant action COUNT; any auditor-risky OR off-window action breaks the grant back to Held (mandatory, not opt-in); the grant lapses immediately if GetLastInputInfo shows the operator went idle or the bound window leaves foreground, forcing a re-tap; auto-grant never covers a re-bind or a new window.
- INV-16 (presence is a HARD requirement for desktop CU, never a default no-op): desktop CU (CuDriverHostEnabled/CuDesktopEnabled) refuses to ARM unless PresenceGuard.IsEnabled && IsAvailable (an enrolled Hello/FIDO2 credential exists). The CU-sovereignty gates (BindCuWindow, EnrollLocalAgentHost, ResumeComputerUse) call AuthorizeAsync with a force-presence flag so the 'lock off => proceed' and 'gate null => proceed' branches CANNOT apply to them, and each requires a FRESH tap - the approval cache is keyed on (WeakeningAction, detail-hash) so a cached bind tap can never satisfy enroll or resume (closes the Critical 'presence off by default' and the HIGH 'one tap covers bind+enroll+resume' findings).
- INV-17 (the bind is unforgeable in-process and target-gated): SetActiveWindow for desktop CU is invoked ONLY from the App's own bind handler, with a CuWindowRef freshly minted by Win32WindowProbe.CaptureForeground() inside the current presence-authorized transaction under a one-time capability token issued by the presence gate - never a caller-supplied ref (closes the 'forge a CuWindowRef for an attacker-owned window' HIGH finding). The bind target is gated against the Slice-2 consent-surface deny-list (credential dialogs #32770, password-manager classes, UAC consent): binding a sensitive-class window requires an extra fresh tap + typed confirmation, the agent's bind-REQUEST justification is shown verbatim, and an agent bind-request can never pre-fill or auto-confirm the target. The HUD surfaces which process/window was just bound.
- INV-18 (HUD visibility is re-confirmed per-INPUT by an adversarial occlusion test, on an App-defined turn clock): before EVERY delivered input the CuExecutorPump must obtain a fresh 'HUD visible RIGHT NOW' ack - topmost AND painted within the last frame AND correct monitor AND the banner's own screen rectangle NOT occluded (enumerate windows above it in z-order / sample the framebuffer region, DWM-cloaking aware), not a self-reported 'I set myself topmost' (closes the per-turn-vs-per-input and 'topmost!=visible' findings). 'Turn' is defined by an App-side epoch/timer, never by driver submission cadence, so a driver cannot hold a turn open to skip re-confirmation. Any single failed confirmation fails closed and withholds THIS input, raising a Medium 'HUD could not be confirmed visible - piloting paused'. Full-screen-exclusive/secure-desktop transitions force handoff.
- INV-19 (the tamper canary runs per-round-trip, not only on the heartbeat): DetectTamper on the shared panic/bind MMF is re-checked on EVERY ExecuteAsync round-trip (not just the 2s heartbeat), so an in-process write that forges boundHwnd or clears the panic byte is caught before the next input rather than up to 2s later; the injector's fail-closed seqlock (odd/torn read => HALTED) bounds forged-boundHwnd risk to consistent writes. Sub-heartbeat forgery by a PROCESS_VM_WRITE attacker remains the documented irreducible in-process residual; the per-round-trip canary shrinks the window to one input.
- INV-20 (panic is the channel-independent hard floor; the channel-drop is advisory): PanicController.Halt -> CuBroker.OnPanicHalt (panic-epoch bump + Reject all non-terminal Desktop items) + CuDesktopPanicFloor.Trigger (BlockInput-shielded release-all + TerminateProcess(injector) + watchdog) + KillPilotNow(shim) fires WITHOUT the driver's cooperation. The authoritative floor is CuBroker.SubmitAsync's halted-Block + Claim returning [] while halted, NOT the channel teardown (which races). Each Submit handler front-doors a halted-state immediate reject so a flooding driver cannot churn the prune window; status polling during halt returns terminal 'halted' without resetting retry backoff. Resume is presence-gated, absent from both pipe contracts, and requires a fresh operator launch+bind - a surviving agent cannot reconnect-storm a just-resumed operator.

## Threat model

- IN SCOPE - medium-IL same-user process: cannot be Foreman's launched signed Pilot shim on HOP A (PID/parent/signed-image pin); cannot answer HOP B's challenge without the operator-transferred session secret; cannot self-enroll as a desktop driver (presence-bound enrollment, force-presence, '*' excluded from Desktop); cannot forge a bind (probe-token-minted CuWindowRef only); cannot win the single-instance connect race (disconnect-and-reloop); cannot hide piloting (per-input adversarial occlusion test). Every action it could get approved is still one-window-confined, audited, HUD-announced, INV-5-verified, and panic-killable.
- IN SCOPE - connect-race / pipe-squat: a leaked per-launch pipe name lets a same-user process OpenExisting first; closed by gating EVERY connection (not just the first) and re-looping the server on any failed handshake, plus a HIGH event when a non-enrolled process connects on a fresh name (it knew the name => leak/race).
- IN SCOPE - auditor/approval-fatigue + auto-grant abuse: a compromised-but-bound agent slow-streaming plausible proposals or riding an auto-grant window; bounded by per-action default-Held, the token-bucket rate limiter, capped+idle-revoking auto-grant TTL, mandatory auditor-risky/off-window breakout, the HUD naming the host, and panic. Not eliminated where the operator opted into auto-grant and walked away - capped to a short bounded window.
- IN SCOPE - INV-7 network leak: closed structurally by the named-pipe transport (no port/bind/SMB-grantable ACE) + the unchanged MCP modality=desktop hard-reject + '*' never extending to Desktop. No bind-correctness invariant a future commit can silently break (the reason Design 2's socket was rejected).
- RESIDUAL (documented, out of bounded model) - same-user attacker with PROCESS_VM_WRITE / PROCESS_DUP_HANDLE on Foreman, or a kernel driver forging LLMHF_INJECTED: can flip the panic/bind MMF or impersonate operator take-back. Identical to the conceded 'an attacker who can write Foreman's memory owns Foreman' residual. The per-round-trip canary (INV-19) shrinks the forged-field window to one input; the HARD floor (TerminateProcess+BlockInput) does not depend on the byte.
- RESIDUAL (documented, root-of-trust assumption) - the operator deliberately binds/enrolls a malicious agent: it drives within the audit/confine/HUD/panic envelope (one bound window, every action audited+held-by-default, panickable), degraded but bounded. Same class as choosing which harness to allow-list over MCP. Mitigated by bind-target gating (sensitive-class windows need extra confirmation) so a bad agent cannot easily steer the operator into binding a credential dialog.
- RESIDUAL (documented, build-quality) - on UNSIGNED/dev builds the Pilot shim's integrity anchor is the at-rest PinBinaryAtRest lock, not Authenticode; a swap BEFORE Foreman starts is uncaught (already game-over: install-dir write pre-launch could patch Foreman.exe). Signed release builds close it via the signer match - same posture as the existing sidecar.

## Open questions (need an operator decision)

- HOP B secret transfer UX: should Foreman OFFER to launch the operator's agent (so the session nonce rides an inherited handle, the strongest option) OR only support the paste-a-one-time-secret-into-agent-config path for agents the operator runs independently? The inherited-handle path is materially more secure but constrains how the operator starts their agent. Operator decision on which to ship first (recommend inherited-handle as primary, paste-transfer as a documented fallback).
- Auto-grant default and cap: confirm CuDesktopAutoGrant stays OFF by default and that the capped TTL (recommend 30-60s) + per-grant action count are acceptable, vs. the operator's tolerance for per-action taps during routine multi-step piloting. The security-correct default is per-action Held.
